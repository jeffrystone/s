## **1. Общая концепция**

**Nodes** — децентрализованная самообучающаяся система, основанная на гомеостатическом принципе. Множество автономных узлов стремится минимизировать свой внутренний диссонанс, а эволюция всей системы направляется непрерывным конфликтом противоположностей:

- **Быстрые метрики ↔ дорогой мощный выстрел**
- **Локальный успех узла ↔ глобальная память (шрамы)**
- **Разведка новых областей ↔ эксплуатация лучших**
- **Память о тупиках ↔ возможность переоткрытия**

Система не имеет внешнего учителя, глобальной функции ошибки и фиксированных гиперпараметров. Все веса и бюджеты адаптируются через локальную обратную связь. Дополнительно предусмотрен **режим ручного управления** (human-in-the-loop), позволяющий оператору через 3D-интерфейс вмешиваться в процесс обучения, не нарушая его непрерывности.

## **2. Основные сущности**

### **2.1 Узел (**`Node`**)**

julia

```
mutable struct Node
    id::UInt64
    params::Dict{Symbol, Any}        # гипотеза (набор параметров)
    hp::Float64                      # вычислительная энергия
    mp::Float64                      # энергия скрещивания
    D::Float64                       # диссонанс (0 – покой, 1 – катастрофа)
    mutation_history::Vector{Tuple{Symbol, Float64}}
    mating_history::Vector{Tuple{UInt64, Float64}}
    active::Bool
    mp_frozen::Bool                  # заморозка MP после провала L5
    heavy_shots::Vector{Tuple{Float64, Bool}} # (D_до, успех)
end
```

- `hp` расходуется на собственные действия (выстрелы, инициирование резонанса).
- `mp` расходуется при пассивном участии в скрещивании; замораживается после провала мощного выстрела; может быть разморожено через успешное участие в резонансах.
- Узел с `hp=0` и `mp>0` остаётся в популяции как «спящая генетическая библиотека».
- Узел с `hp=0` и `mp=0` удаляется.

### **2.2 Шрам (**`Scar`**)**

julia

```
struct Scar
    center::Dict{Symbol, Any}
    radius::Float64
    potential::Float64          # 0..1
    fail_level::Int             # 1..5
    decay_rate::Float64
    last_update::UInt64
end
```

- Потенциал затухает по экспоненте: `p(t) = p0 * exp(-decay_rate * (t - last_update))`.
- При успехе узла (`D < 0.2`) в области шрама потенциал обнуляется (разрушение шрама).
- Шрамы от L5 имеют высокий начальный потенциал и медленное затухание; шрамы от L1 – низкий потенциал и быстрое затухание.

### **2.3 Среда (`**Environment`**)**

julia

```
mutable struct Environment
    nodes::Vector{Node}
    scars::Vector{Scar}
    event_queue::PriorityQueue{Event, Float64}
    exploitation_budget::Float64
    exploration_budget::Float64
    metric_weights::Vector{Float64}         # веса L1..L5
    crossover_weights::Dict{Symbol, Float64}
    resonance_memory::Dict{Tuple{UInt64, UInt64}, Float64}
    tick::UInt64
    cold_start::Bool
    manual_events::Channel{Event}           # канал для ручных вмешательств из UI
end
```

### **2.4 События (**`Event`**)**

julia

```
@enum EventType begin
    SHOT
    HEAVY_SHOT
    RESONANCE
    ANALYSIS
    MANUAL   # ручное вмешательство (добавление/удаление узла, принудительный резонанс и т.п.)
end

struct Event
    type::EventType
    node_id::UInt64
    partner_id::UInt64
    priority::Float64
    manual::Bool                 # true, если событие инициировано из UI
    payload::Dict{Symbol, Any}   # дополнительные данные для ручных действий
end
```

## **3. Иерархия методов оценки (L1–L5)**

Абстрактный интерфейс:

julia

```
abstract type MetricLevel end
struct L1 <: MetricLevel end  # эвристики размера/формы
struct L2 <: MetricLevel end  # Bernstein скор
struct L3 <: MetricLevel end  # Murphy-E скор
struct L4 <: MetricLevel end  # мини-просеивание
struct L5 <: MetricLevel end  # мощный выстрел (полный GNFS)
```

Функция оценки (реализуется пользователем):

julia

```
function evaluate(params::Dict{Symbol, Any}, level::Type{<:MetricLevel})::Tuple{Float64, Bool}
    # возвращает (оценка_качества, успех_делители_найдены?) для L5
end
```

Диссонанс узла `D` вычисляется как взвешенная сумма нормированных оценок от разных уровней, причём веса `metric_weights` автономно калибруются (см. п. 4.5 ANALYSIS).

## **4. Главный цикл и обработчики событий**

### **4.1 Планирование**

Приоритеты событий:

- `SHOT`: `(1 - D) * hp * вес_уровня`
- `HEAVY_SHOT`: как SHOT, с высоким порогом по `hp` и низким `D`
- `RESONANCE`: `mutual_suffering * (mp1+mp2) * complementarity(node1, node2)`
- `ANALYSIS`: фиксированный малый квант, гарантированно каждые `K` тактов
- `MANUAL`: всегда максимальный приоритет (999.0), обрабатывается немедленно

Среда на каждом такте извлекает событие из очереди (или из канала ручных событий) и выполняет соответствующий обработчик.

### **4.2 SHOT (обычный выстрел)**

1. Выбор доступного уровня `L` по `hp` и `D`.
2. `evaluate(params, L)` → обновление `D`, истории мутаций, `hp -= cost[L]`.
3. Если результат неожиданно плох (L3 предсказывал низкий D, а L4 дал высокий) – **апелляция**: узел тратит `hp`/`mp` на переоценку уровнем `L+1`. При выигрыше штрафуется вес метрики `L3`, иначе узел теряет репутацию.

### **4.3 RESONANCE (скрещивание)**

1. Активный узел `A` выбирает партнёра `B` (может быть спящим). Выбор управляется обучаемым вниманием:

`score = α·distance(A,B) + β·suffering_similarity + γ·past_resonance_success`
2. Для скрещивания применяется оператор, выбранный по весам `crossover_weights`.
3. Генерация `N` гибридов (N ~ A.hp * [B.mp](http://B.mp)), каждый сразу проходит L1; выжившие – L2.
4. Лучший гибрид с низким `D` получает начальные `hp`, `mp` и добавляется в популяцию.
5. Обновление `mp` родителей, веса оператора, `resonance_memory`.
6. Часть `hp` гибрида перечисляется в **exploration_budget** (налог на успех).

### **4.4 HEAVY_SHOT (мощный выстрел)**

1. Доступен только при `D < D_thresh`, `hp > heavy_shot_cost` и отсутствии сильных шрамов.
2. `evaluate(params, L5)`. При успехе – останов с результатом.
3. При провале:
  - `hp` обнуляется почти до нуля.
  - `mp` замораживается (`mp_frozen = true`).
  - Создаётся `Scar` с `fail_level=5`, `potential=1.0`, медленным затуханием.
  - Узел остаётся в популяции как пассивная библиотека; может восстанавливаться, участвуя в скрещиваниях за счёт MP партнёров (получая за это небольшой `hp`).

### **4.5 ANALYSIS (фоновый)**

Выполняется каждые `analysis_interval` тактов:

1. **Калибровка метрик**: для выборки узлов, прошедших L3 и L4, вычисляется корреляция. Если L3 систематически завышает качество, `metric_weights[3]` снижается, и наоборот.
2. **Обновление шрамов**: потенциал экспоненциально затухает; шрамы с `potential < ε` удаляются.
3. **Балансировка бюджетов**:
  - Если средний `D` не снижается `stuck_ticks` тактов → `exploration_budget += 10%` за счёт `exploitation_budget`. Принудительная разведка с шумом в механизме внимания.
  - Если прогресс идёт → `exploitation_budget` немного увеличивается.
4. **Удаление мёртвых узлов** (`hp==0 && mp==0 && !mp_frozen`).
5. **Разморозка MP**: если замороженный узел успешно участвовал в нескольких резонансах, `mp_frozen` снимается, `mp` частично восстанавливается.

## **5. Гомеостатические механизмы противоположностей**

- **Комплементарность партнёров**: обучаемое внимание (нейросеть или таблица с обновляемыми весами), которое по эмбеддингам и истории страданий предсказывает успех резонанса. При застое добавляется шум, чтобы система пробовала новые, возможно, «противоположные» пары.
- **Апелляция**: локальный опыт узла противопоставляется глобальной метрике. Узел тратит ресурсы, чтобы оспорить справедливость оценки. В случае успеха метрика корректируется.
- **Разрушение шрамов**: узел с `D < 0.2` в области шрама аннулирует его потенциал – победа настоящего над прошлым.
- **Бюджетная война**: налог на успех подпитывает разведку; при застое – принудительная разведка с шумом. Это противостояние эксплуатации и исследования не даёт системе выродиться.

## **7. Начальная настройка и холодный старт**

### **7.1 Начальная конфигурация**

Задаётся вручную (файл конфигурации или код):

- Диапазоны параметров (например, ведущий коэффициент, границы факторной базы).
- Число начальных узлов `N_init`.
- Начальные энергии `hp=100`, `mp=50`.
- Стоимости методов `cost = [1, 5, 20, 100, 500]`.
- Начальные веса метрик (равномерные или экспертные).
- Пороги для мощного выстрела (`D_thresh`, `hp_thresh`).
- Периодичность анализа `analysis_interval`.

### **7.2 Генерация первой популяции**

Вызывается `generate_random_params(scars)` для каждого из `N_init` узлов. Шрамов в начале нет, поэтому узлы случайно распределены в допустимом пространстве. Дополнительно генератор может создавать несколько «экстремальных» узлов на границах, чтобы сразу охватить противоположности.

### **7.3 Режим холодного старта**

На первых `cold_start_ticks` тактах:

- L1 и L2 бесплатны (HP не тратится) для быстрого накопления статистики.
- Резонансы отключены.
- Шрамы не записываются.
- Бюджет разведки = 100%.

После холодного старта включаются все механизмы, и система переходит в автономный режим.

## **8. Оптимизации для Julia**

- **Типы**: `mutable struct` с нативной типизацией; внутри задач – специализированные структуры вместо `Dict` для параметров.
- **Инкрементальные вычисления** `D` (обновляются только изменившиеся компоненты).
- **Кэш** `evaluate` для одинаковых параметров и уровней.
- **Геометрия**: `SVector` из `StaticArrays` для эмбеддингов и расстояний.
- **Поиск**: `KDTree` для шрамов и партнёров.
- **Параллелизм**: `@threads` / `@spawn` для независимых выстрелов и гибридов; `Channel` для очереди событий и ручных вмешательств.

## **9. UX и 3D-визуализация (WebGL) с ручным управлением**

### **9.1 Общие требования**

Веб-интерфейс, подключённый к Julia-серверу по WebSocket, отображает:

- 3D-сцену с узлами, лучами, шрамами и «Кожей».
- Панели с глобальными метриками и графиками.
- Распределение вычислительных ресурсов (CPU).
- **Интерактивные элементы управления** для ручного вмешательства в ход обучения.

### **9.2 Технический стек**

- **Julia backend**: `HTTP.jl` + `WebSockets.jl` (или `JSServe.jl` для интеграции с `three.js`).
- **Клиент**: браузер, библиотека `three.js`.
- **Протокол**: сервер периодически (каждые 100 мс) отправляет JSON с состояниями узлов, шрамов, событий и метрик. Клиент интерполирует анимации. Обратно отправляются команды ручного управления.

### **9.3 Отображение узлов**

Сфера в 3D-пространстве эмбеддингов (или нормализованных параметров). Кодирование:

- **Размер**: пропорционален `hp` (спящие – маленькие, полупрозрачные).
- **Цвет**: `D` по шкале зелёный (0) → жёлтый (0.5) → красный (1.0).
- **Обводка/свечение**: яркость отражает `mp` (яркое – высокое, серое – замороженное).
- **Статус**: сплошная сфера (активный), с трещиной (заморожен), полупрозрачная (спящий).

### **9.4 Анимации событий**

- **SHOT**: тонкая линия от узла, цвет по уровню (L1 серый … L4 фиолетовый), затухает.
- **HEAVY_SHOT**: яркий луч с искрами; при провале – расширяющаяся красная сфера, превращающаяся в шрам.
- **RESONANCE**: вспышка между партнёрами, веер нитей к выжившим гибридам (золотой цвет).

### **9.5 Визуализация шрамов**

Полупрозрачные эллипсоиды от тёмно-красного (свежий L5) до бледно-розового (старый L1). Прозрачность пропорциональна `potential`. При разрушении – анимация лопающегося пузыря.

### **9.6 Глобальные индикаторы и графики**

- **Верхняя панель**: средний D, бюджеты (exploration vs exploitation), число узлов (всего/активных), число резонансов, лучший D.
- **График времени**: линия среднего D и полосы бюджета по тактам.
- **Текстовый блок**: параметры текущего лидера.

### **9.7 Мониторинг ресурсов процессора**

- **Тепловая карта ядер**: квадраты по числу логических ядер, цвет от зелёного (низкая загрузка) до красного (высокая).
- **Индикаторы на узлах**: вертикальный столбик над узлом, показывающий выделенное процессорное время (пропорционально `hp`, потраченному за последний интервал). При `hp=0` исчезает.
- **Круговая диаграмма событий**: распределение времени между SHOT, RESONANCE, HEAVY_SHOT, ANALYSIS, MANUAL.
- **График использования**: линия общей загрузки CPU и разбивка по типам событий.

Сервер включает в снапшот метрики: `cpu_usage_per_core`, `event_times`, `per_node_time`.

### **9.8 Ручное управление (Human-in-the-loop)**

Все ручные действия передаются с клиента на сервер как события с `type=MANUAL` и максимальным приоритетом. Они сразу попадают в очередь и обрабатываются как обычные события, но не расходуют бюджеты автоматически (может быть настроено).

Оператор может:

- **Добавить узел**: клик правой кнопкой в сцене → «Создать узел». Можно ввести параметры или выбрать случайные. Узел появляется в популяции с начальными `hp`, `mp`, `D=0.5`.
- **Удалить узел**: клик по узлу → «Удалить». Узел убирается, но перед этим записывается шрам, чтобы предотвратить немедленное перерождение точно такого же.
- **Принудительный резонанс**: выделить два узла (Ctrl+клик) → «Инициировать резонанс». Среда немедленно выполняет скрещивание с наивысшим приоритетом. Результат записывается как обычно.
- **Сдвиг противоположностей**: выбрать два узла и команду «Назначить эталонной парой». Веса механизма внимания для схожих пар увеличиваются, а глобальные параметры `α, β, γ` корректируются, чтобы поощрять такие расстояния и паттерны страданий. Это задаёт направление поиска.
- **Аннулировать шрам**: клик по шраму → «Убрать». Потенциал шрама мгновенно обнуляется, открывая запретную область.
- **Заморозить/разморозить узел**: принудительно заморозить `mp` или разморозить (например, вернуть в активную фазу).

Все действия логируются, а их влияние (успех или неудача) отслеживается. Если ручное вмешательство систематически приводит к успеху, соответствующие веса механизмов (например, внимание) обучаются в пользу таких вмешательств. Если нет – система со временем «переварит» их через шрамы.

### **9.9 Интерактивность камеры**

- Вращение, масштабирование, перемещение.
- При наведении на узел – всплывающая подсказка с id, D, HP, MP, параметрами и историей.
- Клик по узлу подсвечивает его и связанные шрамы/резонансы.
- Фильтры слоёв: включение/отключение шрамов, лучей, спящих узлов.
- Пауза/замедление симуляции.
